### ADR: Поэтапный план декомпозиции монолита GoFuture на микросервисы

### Автор: Терехин А.В.
### Дата: 07.03.2026

## Контекст

Компания GoFuture — агрегатор такси с монолитной архитектурой на Django, которая не справляется с ростом нагрузки при экспансии в Юго-Восточную Азию и Южную Америку. В пиковые периоды (свыше 100 тыс. активных поездок) происходят каскадные отказы, разработка новых функций занимает полгода и более, а восстановление после сбоев — свыше 4 часов.

**Бизнес требует:**
- Обрабатывать 500 тыс. конкурентных поездок
- Обеспечить 99,95% доступность
- Сократить время вывода функций до 2 недель
- Поддерживать динамическое ценообразование и умное распределение водителей
- Соответствовать локальным регуляторным требованиям

Решение — поэтапный переход от монолита к микросервисной архитектуре с выделением доменных сервисов, соответствующих продуктовым командам.

## Требования

| № | Требование | Описание |
| --- | --- | --- |
| 1 | Управление поездками | Создание, отслеживание, завершение поездок в реальном времени |
| 2 | Динамическое ценообразование | Расчёт стоимости поездки на основе спроса, предложения, времени суток и геозоны |
| 3 | Умное распределение водителей | Назначение не ближайшего, а оптимального водителя для предотвращения «горячих зон» |
| 4 | Геопоиск и маршрутизация | Интеграция с картографическими сервисами для поиска водителей и расчёта маршрутов |
| 5 | Обработка платежей | Приём оплаты от пассажиров через локальные платёжные шлюзы (Яндекс Пэй, банки и др.) |
| 6 | Выплаты водителям | Автоматические выплаты на банковские счета с учётом комиссий и налогов |
| 7 | Уведомления | Push-уведомления (FCM, APNs, Huawei Push) о статусе поездки, назначении водителя и т.д. |
| 8 | Антифрод | Анализ транзакций и поведения пользователей для выявления мошенничества |
| 9 | Корпоративное управление | Учёт корпоративных поездок, выставление счётов, управление бюджетами |
| 10 | Мультирегиональность | Поддержка локальных требований: валюты, языки, регуляторика, платёжные системы |
| 11 | API для партнёров | Самообслуживание через внешние API для интеграции с локальными сервисами |
| 12 | Аналитика в реальном времени | Сбор событий, прогнозирование спроса, ML-модели для оптимизации бизнеса |

## Нефункциональные требования

| Категория | Требование | Целевое значение | Метрика измерения |
| --- | --- | --- | --- |
| Производительность | Обработка конкурентных поездок | ≥ 500 000 одновременных поездок | Транзакции/сек в пик |
| Доступность | Критические сервисы (Booking, Driver) | 99,95% SLA | Время простоя/месяц ≤ 22 мин |
| Масштабируемость | Горизонтальное масштабирование | Автоматическое в 3+ регионах | Время масштабирования < 5 мин |
| Время разработки | Выход новых функций на рынок | ≤ 2 недели | Cycle time от идеи до продакшена |
| Надёжность | Время восстановления (MTTR) | ≤ 1 час | Среднее время устранения инцидентов |
| Наблюдаемость | Покрытие критических метрик | 100% | Количество неохваченных метрик = 0 |
| Согласованность | Стратегия распределённых транзакций | Eventual consistency + Saga | Время согласования данных < 30 сек |
| Безопасность | Соответствие регуляторным требованиям | 100% по регионам | Количество нарушений = 0 |
| Отказоустойчивость | Изоляция сбоев | Отказ одного сервиса ≠ каскадный отказ | Количество затронутых сервисов при сбое = 1 |

## Решение

### Карта сервисов

| Домен | Сервис | Ответственная команда | Ключевые сущности | Внешние зависимости |
| --- | --- | --- | --- | --- |
| Идентификация | identity-service | Платформенная | Пользователи, сессии, права | — |
| Бронирование | booking-service | Booking | Поездки, заказы, статусы | driver, pricing, fraud |
| Водители | driver-service | Driver | Профили, геолокация, статусы | geography |
| Ценообразование | pricing-service | Pricing | Тарифы, динамические цены, скидки | geography, booking |
| Платежи | payments-service | Payments | Транзакции, шлюзы, возвраты | booking, identity |
| Выплаты | payouts-service | Payments | Балансы водителей, выплаты | driver, bank-api |
| Уведомления | notification-service | Notification | Шаблоны, каналы, доставка | booking, driver |
| География | geography-service | Geography | Зоны, маршруты, "горячие точки" | maps-api |
| Фрод | fraud-service | Fraud | Правила, скоринг, инциденты | booking, payments |
| Аналитика | analytics-service | Analytics | События, агрегаты, метрики | Все сервисы (через события) |
| Шлюз | api-gateway | Платформенная | Маршрутизация, рейт-лимиты | Все сервисы |

### Выделение сервисов по приоритетам

1 этап
| Приоритет | Сервис | Обоснование |
| --- | --- | --- |
| Критический | identity-service | Низкая связанность с бизнес-логикой<br> Критично для безопасности и регуляторных требований<br> Простая миграция (таблица users изолирована) |
| Высокий | notification-service | Полностью асинхронный (через очередь)<br> Нет транзакционных зависимостей<br> Можно выделить через dual write без простоя |

2 этап
| Приоритет | Сервис | Обоснование |
| --- | --- | --- |
| Высокий | driver-service | Высокая нагрузка в пик (геолокация)<br> Ключевой для масштабируемости по регионам<br> Требует тесной интеграции с geography → выделять парами |
| Высокий | geography-service | Для реализации "умного" распределения водителей<br> Требует выделения до booking (зависимость "водитель → геозона") |
| Средний | payouts-service | Финансово критичный, но изолированный домен (балансы водителей)<br> Можно выделить через CDC из payments |

3 этап
| Приоритет | Сервис | Обоснование |
| --- | --- | --- |
| Средний | pricing-service | Высокая связанность с booking и geography<br> Требует стабильных зависимостей перед выделением |
| Средний | payments-service | Регуляторная сложность (PCI DSS)<br> Требует тщательного тестирования перед миграцией |
| Низкий | fraud-service | Зависит от данных из нескольких источников<br> Можно выделить после стабилизации основных сервисов |

4 этап
| Приоритет | Сервис | Обоснование |
| --- | --- | --- |
| Низкий | booking-service | Ядро бизнес-логики с максимальной связанностью<br> Выделять в конце, когда все зависимости уже микросервисы |
| Низкий | analytics-service | Read-only, можно клонировать данные без риска |


### Обеспечение обратной совместимости

| Механизм | Применение | Срок действия |
| --- | --- | --- |
| Паттерн Strangler Fig  | Прокси перенаправляет запросы к монолиту или микросервису в зависимости от пути | До полного вывода монолита (12 месяцев) |
| Dual Write | Запись в БД монолита + новый сервис одновременно | 1-3 месяца на этапе миграции |
| CDC (Change Data Capture) | Debezium захватывает изменения из БД монолита для синхронизации | 2-4 месяца на этапе миграции |
| API Versioning | Поддержка /v1/ (монолит) и /v2/ (микросервисы) одновременно | 6 месяцев после миграции |
| Feature Flags | Постепенное включение нового функционала для части пользователей | Постоянно (для безопасного развёртывания) |
| Событийная синхронизация | Публикация событий из монолита в Kafka для потребления микросервисами | До вывода монолита |

### План миграции данных

| Сервис | Объём данных | Стратегия | Этапы | Риски и митигация |
| --- | --- | --- | --- | --- |
| identity-service | 5 ГБ | Big Bang | 1. Выделить таблицу users<br>2. Настроить репликацию на 1 час до миграции<br>3. Переключить трафик за 5 минут | Риск: потеря сессий <br> Митигация: сохранять сессии в Redis с TTL |
| notification-service | 2 ГБ | Событийная миграция | 1. Запустить воркеры на новом сервисе параллельно монолиту<br>2. Постепенно переключать каналы уведомлений | Риск: дублирование уведомлений <br>Митигация: идемпотентность по event_id |
| driver-service | 50 ГБ | Шардирование по регионам | 1. Выделить геоданные в отдельную БД с шардированием по region_id<br>2. Мигрировать регионы по одному (Европа <br> Азия <br> Америка)<br>3. CDC для синхронизации в реальном времени | Риск: несогласованность геопозиций <br> Митигация: двойная проверка через geography-service |
| geography-service | 15 ГБ | Поэтапная миграция | 1. Выделить таблицы zones, hotspots<br>2. Dual write на 2 недели для валидации<br>3. Отключить запись в монолит | Риск: расхождение в расчёте "горячих точек" <br> Митигация: сравнение результатов алгоритмов |
| payments-service | 200 ГБ | Шардирование + строгая согласованность | 1. Выделить транзакции в отдельную БД с шардированием по merchant_id<br>2. Использовать распределённые транзакции (Saga) для согласованности | Риск: двойной платёж <br> Митигация: идемпотентность по payment_id + компенсирующие транзакции |
| payouts-service | 80 ГБ | Поэтапная миграция с валидацией | 1. Выделить таблицы driver_balances, payouts<br>2. Dual write + ежедневная сверка балансов<br>3. Переключение после 7 дней без расхождений | Риск: расхождение балансов <br> Митигация: автоматическая коррекция через adjustment транзакции |
| booking-service | 500 ГБ | Шардирование по регионам + по времени | 1. Выделить исторические данные (> 90 дней) в архивную БД<br>2. Шардировать активные поездки по region_id<br>3. CDC для синхронизации в реальном времени | Риск: потеря данных при миграции <br> Митигация: точечные бэкапы перед каждым этапом |
